### Question 2
> How do you distinguish a UX problem from a business problem?

**Answer:**
yeah the ux problem is centered around usability, around cognitive load, around accessibility, around user friction. for example users can't understand how to customize the product. the business problem is about customer acquisition cost, customer monthly churn percentage, conversion rate, etc.

## PRODUCT STRATEGY

### Question 6
> How do you define an MVP?

**Answer:**
mvp is a convention that we take into consideration, taking the variables that we need to adjust. we take time into consideration and a measurable amount of value that we will deliver at a certain point in time. we are able to dissect this deliverable and organize the team too so we can tackle everything around the product. so obviously it's like a minimal viable product right? we try to maximize value and then keep the effort controlled.

---

### Question 7
> What is NOT an MVP?

**Answer:**
yeah what is non-mvp? it's a really complete set of features that at a certain point in time we are not able to measure the impact. we assume or we have certain information that those features are not delivering the same value as the features and the actual core of the product we are delivering so we need to be pretty cautious about this.

in terms of things that are obvious for example:
- super polished aesthetics
- a ui that is glamorous and heavy in the web browser
- a lot of things that are obviously not part of the mvp


also things that could live in the limits of being a feature that deserves to be in an mvp and a feature that is not going to be deserving to be in an mvp. how we can decide that is the expertise of the subject matter expert. what product are we building and our stakeholders will be able to help us to identify that. it depends on the product. we will be able to understand what we are doing in terms of let's say:
- uber: we are letting people contact, ask for a ride, pay for the ride, and then go wherever they are going to


rather than a lot of features like:
- sharing your location with friends
- splitting the bill
- recording this


so it's a lot of tools that are around the most important core value.

after analyzing all the features with all the metrics, we make sure that we try to come up with some sort of a system. we brainstorm a system to identify which are the priorities. when we are prioritizing, we need to make sure that we evaluate the effort from the engineering front and then we pick what we want to build with a stakeholder.

## DISCOVERY

### Question 11
> What is Product Discovery?

**Answer:**
product discovery is when we test what we know about everything that we are trying to build. we have a lot of assumptions, a lot of content, and a lot of expertise in the area ideally. we identify pain points and then we propose solutions. what we are doing is trying to understand if those solutions that we propose are aligned with the pain points that we identify.

---

### Question 12
> Describe your discovery process from start to finish.

**Answer:**
yeah so i usually start with the kick-off session when i want to raise my voice to identify the problem definition. that has to be in all people's minds because we need to be aligned at the beginning on the one thing that we know is a fact and that we believe is true. then we start navigating into that, going deeper and analyzing that problem, gathering all the information about it, mapping it, getting data, getting ideation, then proposing solutions with cross-functional teams or partners or everything.

then we need to prototype the solution but actually first of all i think we need to make sure that we understand the process where the process is located. we have a good context diagram then we have a good process diagram. we have a good solution for the portion of the process that we are tackling or step or portion of the process that we are tackling, which depends on the product of course. then we start prototyping, validation, usability, everything else right and then it ends up with a handoff mapping of the things that we discover and implement and design.

### Question 13
> A stakeholder requests a new feature.

> What happens next?

**Answer:**
then i'll pick a framework to evaluate the feature in terms of what kind of feature i'll be looking for. i decide to go: if i need rise or the other models for feature request evaluation then after that i run it through the evaluation and propose some insight about that, a report on that. we make sure that it is well located in the roadmap of all the things that are existing because i have never been in a product that is empty. the work that we need to do is to understand where we have to locate it and then we start the discovery process.

it's like i feel like it's like a regression test. we don't break things that are already working so i want to have that mindset in engineering but also user-centered or oriented. i need to empathize with the user and then we make sure that these new features are going to work for them. we discovered everything about it. we run from:
- what the problem is

- what metrics are going to be improved with this feature


after this we ideate a design solution, prototyping, and hand-off design and that's it. 

in my last position we had a platform called aha so we laid out ideas from everybody including stakeholders. we run a process to identify what is the risk associated with it, the value of it. the framework that we picked then is after we decide to gather more information or we have seven or eight stages for it. after we run all the stages then we make sure that that feature is ready to go right into the backlog on jira and that's pretty much it.

### Question 14
> How do you validate assumptions?

**Answer:**
i think an assumption could become a fact when you validate it with data right? the source of the data will be attached to the nature of the assumption. if the assumption is a result of an observation that we are having from analyzing data, then it will be way easier to tackle.

an assumption that is coming from the highest-paid person in the company that says "hey i think users want to perform task b not task k" could be right but we need to find data that backs up that claim. we need to use common sense. no we need to classify that assumption:
- what is the uncertainty, the level of entropy of that assumption or uncertainty?
- how uncertain is that assumption?
- what's the feasibility generally speaking when we understand that assumption, is it feasible or not feasible?

### Question 15
> How do you know when you have enough evidence to move forward?

**Answer:**
in my experience time and effort are the constraint. in a perfect world i will aim for 99.99% of accuracy, backing up an assumption or a hypothesis or a feature or whatever we want to build. i never reach that level of confidence but we need to agree. what is the risk associated with lowering that level of confidence and can we handle that risk or not? we can aim for 95%. the risk will be way lower than if we aim for 65% and we can calculate that based on the frame. we can pick a couple of frameworks to do that. i'm not sure which but i'm sure that there are some frameworks to tackle the accuracy of the evidence that i have for something.
