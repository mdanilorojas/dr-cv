Floor AAccessibility · WCAG AA minimum, AAA aspirational #

Reference set audited mentally: emilkowal.ski, rauno.me, paco.me, brittanychiang.com, leerob.com, linear.app, vercel.com, stripe.com, anthropic.com. Consensus floor is higher than many designers assume. Brittany Chiang ships AAA contrast. Rauno's site honors reduced-motion on every animation. Linear's landmark structure is exhaustive. Stripe ships focus rings you could etch on glass.

What the floor is

The landing must be fully operable by keyboard alone, screen-reader coherent, and visually usable at WCAG AA contrast, with reduced-motion honored.

Concretely: every interactive element reachable by Tab, a visible focus ring at all times, semantic landmarks (<header> / <main> / <nav> / <footer>) so screen readers can jump, a skip link for long hero sections, text contrast ≥ 4.5:1 (body) and ≥ 3:1 (large text, UI), decorative imagery marked aria-hidden="true", meaningful alt="" on content images, and all motion disabled when prefers-reduced-motion: reduce. These are not "nice to have" — they are the defaults at Stripe, Linear, Vercel, Anthropic.

Specific test · the 12-item a11y checklist
  1. A1Tab through every link and button. Focus ring must be visible on every stop, never clipped by overflow:hidden, never under a sticky header. Press Shift+Tab — reverse order must match visual order.
  2. A2Skip link present. First tab stop on page is <a href="#main">Skip to content</a>, visually hidden until focused. Hero is long; skip link is mandatory.
  3. A3Landmarks cover the page. One <header>, one <main>, one <footer>, one or more <nav> with aria-label. VoiceOver rotor (VO+U) / NVDA landmark list should expose them.
  4. A4Heading order is monotonic. One h1. No h3 before h2. Check with devtools Accessibility tab.
  5. A5Color contrast ≥ 4.5:1 for body text, ≥ 3:1 for 18px+ or bold 14px+. Run all text/background pairs. Light gray on dark background is where portfolios fail — the --hl-text-60 token (#95979E on #090A0C) = 6.8:1 ✓; --hl-text-50 (#797D86) = 4.8:1 ✓ at body size; --hl-text-40 (#61656B) = 3.2:1 ✗ for body — legal only for large text.
  6. A6Focus ring has ≥ 3:1 contrast against adjacent colors (WCAG 2.4.11). Orange #FF8964 2px ring on black = 7.4:1 ✓; on cream section (#F6F4F0) = 2.6:1 ✗ — cream section needs a different ring color.
  7. A7Images have alt text or are explicitly decorative. Every <img> needs alt="..." (content) or alt="" (decorative). SVG diagrams need role="img" + aria-label. No <img> without alt, ever.
  8. A8Decorative canvas / particle layers are aria-hidden="true" and role="presentation". The particle beam in the hero is pure decoration — screen readers must ignore it.
  9. A9prefers-reduced-motion: reduce disables every animation. Not just transitions — also the particle canvas, the marquee, the CTA glow, the reveal-on-scroll. Reviewers with vestibular sensitivity will test this.
  10. A10No role="marquee". It's an invalid ARIA role in the 2025 spec and screen readers announce nothing useful. Use a plain <ul> with aria-hidden="true" if purely decorative, or aria-label if informational.
  11. A11Language attribute is correct per route. <html lang="en"> on /, <html lang="es"> on /es/. Any inline foreign phrase wrapped in <span lang="...">.
  12. A12Form inputs (if any) have labels. Current landing has no forms — keep it that way for V11. mailto: + copy-email pattern is lower cognitive load.

Floor BPerformance · Lighthouse 90 / 95 / 100 / 100 and Core Web Vitals #

What the floor is

Lighthouse ≥ 90 Performance, ≥ 95 Accessibility, 100 Best Practices, 100 SEO on Mobile throttled (Slow 4G, Moto G4 class).

Tier-1 interviewers run Lighthouse from devtools on a mobile preset before reading a word. Desktop scores are easy — mobile is the real test. Core Web Vitals targets: LCP < 2.0s (tier-1 portfolios run ~1.0–1.5s; the 2.5s "good" line is too low for a static portfolio), INP < 100ms, CLS < 0.05. Total JS budget for a senior IC portfolio sits around 15–30 KB gzipped. Above 100 KB of JS for a static portfolio reads as lack of taste.

Specific test · the 8-item perf checklist
  1. B1Run Lighthouse mobile preset on deployed URL. Local file:// runs lie — test the live URL. Four scores: target Perf ≥ 90 · A11y ≥ 95 · BP 100 · SEO 100.
  2. B2LCP element identified and optimized. Open Lighthouse Treemap — LCP is likely the hero h1 text or a hero image. Text LCP needs the font loaded with font-display: swap and <link rel="preload" as="font" crossorigin> for the primary weight.
  3. B3Total transfer < 200 KB for a single-file portfolio. HTML + inline CSS + one JS block + fonts. Reference: paco.me ≈ 40 KB, rauno.me ≈ 90 KB, leerob.com ≈ 180 KB.
  4. B4JS budget < 30 KB gzipped. No framework runtime on a static portfolio. Vanilla IIFE is fine. Zero third-party scripts except optional analytics (< 5 KB).
  5. B5No render-blocking resources. CSS inlined in <head>, JS deferred or at end of <body>, fonts loaded with display: swap.
  6. B6CLS < 0.05. Fonts must not reflow the hero headline — use size-adjust, ascent-override, or match system-stack metrics. Images must have explicit width / height.
  7. B7INP < 100ms. Particle canvas must not block interaction — use requestAnimationFrame, not setInterval, and pause when hero is offscreen (already done, good). No scroll-jacking.
  8. B8HTTP caching + compression on the host. Brotli/gzip on HTML + CSS + JS. Cache-Control: public, max-age=31536000, immutable on fingerprinted assets.

Floor Ci18n / bilingual discipline · EN primary, ES toggle #

What the floor is

Two real languages. Two real URLs. One non-negotiable: human-written Spanish, no machine translation, no Spanglish.

Danilo is bilingual. The ES version must be written by him, not translated at build time. Typography must handle ñ ¿ ¡ á é í ó ú ü without tofu or clipped diacritics. Numbers and dates must format per locale. Language switcher must preserve scroll position and section.

Specific test · the 8-item i18n checklist
  1. C1URL structure: path-based, not query-string. /es/ is correct. ?lang=es is worse for SEO (some crawlers treat it as duplicate content) and fragile to query-string stripping.
  2. C2hreflang pairs are reciprocal and absolute. On /: <link rel="alternate" hreflang="en" href="https://danilorojas.xyz/"> and hreflang="es" href="https://danilorojas.xyz/es/" and hreflang="x-default" href="https://danilorojas.xyz/". Both pages must reference both URLs.
  3. C3<html lang> matches route. lang="en" on /, lang="es" on /es/. Wrap isolated foreign phrases (e.g., "Agentic Designer" when it stays English in Spanish copy) in <span lang="en">.
  4. C4Language switcher preserves section. Clicking ES on /#method goes to /es/#method, not to /es/. Test with anchor links.
  5. C5Typography test string renders correctly: "¿Construyendo una interfaz para agentes? El año pasado logré entregar más rápido: niño, señor, añil." — ñ with tilde properly centered, ¿¡ present, no missing glyphs.
  6. C6Dates / numbers formatted per locale. "May 7, 2026" → "7 de mayo de 2026". "3.4M users" → "3,4M usuarios" (comma as decimal in es-ES; es-419 may use period). Pick one ES variant (Danilo is Ecuadorian — es-EC / es-419) and stick to it.
  7. C7Content parity, not translation parity. ES version can omit sections that don't translate (an English pun, for example) and add a section that's only culturally relevant in LATAM. "Parity of intent" > "word-for-word equivalent".
  8. C8No automatic browser-language redirect. User chose the URL. Respect the choice. Offer the toggle; don't redirect.
How dr-cv currently compares
  • passPath-based routing: /es/ exists (dist/landing-huly/es/). Correct architecture.
  • passlang attribute correct per route. lang="en" on root.
  • partialhreflang present but relative paths. Current: <link rel="alternate" hreflang="es" href="/es/">. Google's guidance: use absolute URLs. Not a hard fail but not tier-1 polish.
  • failNo x-default. Missing <link rel="alternate" hreflang="x-default" href="/">. Needed so Google knows what to show users with unsupported languages.
  • failLanguage switcher is a hard link to /es/. Does not preserve hash/section. User on /#method who clicks ES lands at /es/ top — loses context. Needs a tiny script: href = '/es/' + location.hash.
  • partialES content exists (per README: "Single-file tab-SPA per language"). Danilo wrote it (not machine-translated) per project memory. Good. Needs spot-check for Spanglish.
  • failDate/number formatting not audited. Unknown whether "2026-05-07" becomes "7 may 2026" in ES or stays ISO.
What to fix
  1. Absolute hreflang + x-default on both pages: 
    <link rel="alternate" hreflang="en" href="https://danilorojas.xyz/">
<link rel="alternate" hreflang="es" href="https://danilorojas.xyz/es/">
<link rel="alternate" hreflang="x-default" href="https://danilorojas.xyz/">
  2. Hash-preserving language toggle:

  a.addEventListener('click', function(e){
    e.preventDefault();
    var target = a.dataset.langSwitch; // "/" or "/es/"

  4. Typography smoke test string included in the test suite: a hidden element containing the full Spanish glyph set; a Playwright / Puppeteer test asserts it renders without overflow.
  5. Add the target domain (danilorojas.xyz or whatever is chosen) to landing.yaml as a build-time constant so absolute URLs can be generated everywhere.

Floor DMotion discipline · for an Agentic UX portfolio #

What the floor is

Every animation must (1) serve information or feedback, not decorate; (2) honor prefers-reduced-motion completely; (3) stay under 300ms for transitions, under 600ms for "intro" reveals; (4) pause when offscreen; (5) not fight the user's scroll.

Emil Kowalski, Rauno, and Paco have all written publicly that "motion is a junior signal when it's decorative and a senior signal when it's informational." For an Agentic UX portfolio, motion should demonstrate taste: restraint, purpose, and reduced-motion hygiene.

Specific test · the 7-item motion checklist
  1. D1prefers-reduced-motion: reduce flips every animation off. Not to "shorter duration" — to zero. No parallax, no particles, no slide-in, no scroll-jacking.
  2. D2Durations follow a scale. 150ms (micro: hover, press), 200ms (small: color, opacity), 300ms (medium: layout), 500ms (large: reveal, modal). No 1234ms custom values.
  3. D3Easing is cubic-bezier(0.4, 0, 0.2, 1) for in-out, cubic-bezier(0, 0, 0.2, 1) for enter, cubic-bezier(0.4, 0, 1, 1) for exit. Linear is for progress indicators only. ease is lazy.
  4. D4Offscreen animations are paused. IntersectionObserver on the particle canvas / any looping animation. Battery-aware.
  5. D5Animate transform and opacity only. Never top, left, width, height, margin — they trigger layout on every frame.
  6. D6No scroll-linked animations without will-change restraint. Scroll-driven opacity is fine; scroll-driven 3D transforms on a 60fps budget is where jank appears.
  7. D7CSS transitions > JS animations for everything except canvas / WebGL. Browser-optimized paths are faster and respect reduced-motion automatically via the media query.
How dr-cv currently compares
  • passReduced-motion tokens zeroed: @media (prefers-reduced-motion: reduce) { :root { --hl-dur-fast: 0ms; --hl-dur-med: 0ms; --hl-dur-slow: 0ms; } }. Excellent — all CSS transitions collapse.
  • passParticle canvas gated on reduceMotion flag in JS. Not even spawned when user prefers reduced motion.
  • passReveal-on-scroll observer also gated on reduceMotion. Elements just appear.
  • passParticle canvas pauses offscreen via IntersectionObserver.
  • passDuration scale defined (--hl-dur-fast 200ms · med 300ms · slow 500ms) and used consistently.
  • passEasing token --hl-ease: cubic-bezier(0.4, 0, 0.2, 1) used throughout.
  • partialMarquee animation — this is the weakest link. Infinite scroll marquees are typically pure decoration; they should be entirely disabled under reduced-motion. Need to check if the CSS animation on .hl-marquee__track is tied to the zeroed duration tokens or has a hardcoded duration.
  • partialCTA cursor-follow glow — runs on pointermove, no throttle. On low-end devices this fires 120+ times per second. Should be requestAnimationFrame-throttled and skipped on coarse pointers (@media (pointer: coarse)).
What to fix
  1. Gate the marquee animation on reduced-motion explicitly: 
    @media (prefers-reduced-motion: reduce) {
  .hl-marquee__track { animation: none; }
}
  2. Skip cursor-follow glow on coarse pointers (touch devices don't have a cursor to follow): 
    if (matchMedia('(pointer: coarse)').matches) return;
  3. rAF-throttle the pointer glow: 
    var pending = false, lastE;
cta.addEventListener('pointermove', function(e){
  lastE = e;
  if (pending) return;
  pending = true;
  requestAnimationFrame(function(){
    // read lastE and update CSS vars
    pending = false;
  });
});
  4. Document the motion budget in README — "3 tokens, 1 easing, every animation auto-disabled on reduced-motion" is itself a taste signal.

Floor ETypography discipline · self-host, subset, variable #

What the floor is

Self-hosted variable fonts, Latin-1 subsetted, preloaded for the LCP weight, with a metric-matched fallback that produces < 0.02 CLS.

Google Fonts CDN is a tier-1 comp liability for three reasons: (1) third-party request adds 100–400ms latency, (2) German/EU GDPR rulings have fined sites for leaking IPs to Google, (3) reviewer sees fonts.googleapis.com in the Network panel and reads "didn't care enough to self-host". Variable fonts are an actively positive signal — they show the designer knows their type.

Specific test · the 6-item typo checklist
  1. E1No third-party font CDN in Network panel. All fonts served from the same origin as HTML. fonts.googleapis.com, use.typekit.net, fonts.bunny.net — all fail tier-1. (Bunny is a lesser failure but still third-party.)
  2. E2Variable fonts where available. Inter VAR is ~120 KB for the full Latin-1 subset and replaces 8 static weights totaling ~600 KB. Reviewer sees font-weight: 100 900 in DevTools and reads "they know".
  3. E3Latin-1 subset. If the site doesn't show Cyrillic / Greek / Arabic, don't ship them. unicode-range: U+0000-00FF, U+0131, U+0152-0153, U+02BB-02BC → ~35 KB WOFF2 per variable font.
  4. E4LCP font preloaded. <link rel="preload" as="font" crossorigin> for the weight used in the hero h1.
  5. E5Metric-matched fallback for < 0.02 CLS. size-adjust + ascent-override + descent-override tuned so the system font renders at the same visual height as the web font.
  6. E6OpenType features active. font-feature-settings: "kern", "liga", "calt" (kerning, standard ligatures, contextual alternates) at minimum. "ss01", "ss02" if using Inter stylistic sets (alt "a", alt "g") — choice itself is a taste signal.

Floor FSEO / shareability floor · OG image, canonical, sitemap #

What the floor is

When a recruiter pastes the URL into Slack/LinkedIn/Twitter, it must render a beautiful card: title, 1200×630 OG image, correct description. When Google crawls it, it finds a canonical URL, a sitemap, and a robots.txt.

The OG image is a designed artifact. For a designer, shipping a landing without a custom OG image is like showing up to a pitch with a blank slide. It's the single most-viewed piece of visual work on the portfolio (every link share), and tier-1 designers invest in it.

Specific test · the 8-item SEO/share checklist
  1. F1Title tag structure. Name — Role | Short hook. Max ~60 chars for Google SERP. Example: "Danilo Rojas — Agentic Designer" (31 chars, good).
  2. F2Meta description. 150–160 chars, first sentence is the hook, no truncation. Unique per page (EN vs ES).
  3. F3Canonical URL. <link rel="canonical" href="https://…"> on every page — absolute. Prevents duplicate-content penalties across / and /index.html.
  4. F4OG image 1200×630, < 300 KB, PNG or JPG (WebP unreliable on Slack). Absolute URL in og:image. Must include og:image:width and og:image:height for fast card rendering.
  5. F5Twitter card. <meta name="twitter:card" content="summary_large_image"> + twitter:image (same as og:image).
  6. F6Structured data (optional but +1). JSON-LD Person schema with name, jobTitle, sameAs (linkedin, github).
  7. F7Sitemap.xml with both language URLs + robots.txt allowing all.
  8. F8Test the card at opengraph.xyz and cards-dev.twitter.com/validator. Don't ship unverified.

  • passTitle + description present and well-formed.
  • passog:title + og:description + og:type present.
  • failNo og:image. Recruiter pastes URL in LinkedIn → blank card or default placeholder. First impression wasted.

Floor GPrivacy / analytics floor · no cookie banner for a portfolio #

What the floor is

A senior IC portfolio ships either no analytics, or privacy-first analytics (no cookies, no GDPR banner). Google Analytics 4 with cookies on a portfolio is an instant taste-fail signal.

The floor at Vercel, Linear, Stripe-engineers' personal sites: Plausible, Fathom, Vercel Analytics, or nothing. None of these require a cookie banner in the EU. Fathom and Plausible are explicit about it on their home pages. A cookie banner on a portfolio signals "I copy-pasted GA without reading the privacy law" — which tier-1 hiring managers read as junior.

Specific test · the 5-item privacy checklist
  1. G1No cookie banner unless the page sets tracking cookies. If you don't set cookies, GDPR doesn't require a banner. Easier to not set cookies.
  2. G2No gtag, no _ga cookie, no fbq, no Hotjar, no Microsoft Clarity in Network panel.
  3. G3If analytics at all: Plausible (€9/mo, EU-hosted, no cookies), Fathom ($14/mo, no cookies), Vercel Analytics (included with hosting, no cookies). All three pass GDPR without a banner.
  4. G4Privacy policy link (optional) if collecting even pseudonymous metrics. One-page, honest.
  5. G5No third-party embeds that set cookies (YouTube default embed, Vimeo default, Twitter embed). Use youtube-nocookie.com, static images, or self-host.
How dr-cv currently compares
  • passZero analytics currently. No third-party scripts. No cookies. No banner needed. Cleanest possible posture.
  • passNo third-party embeds. No YouTube, no Vimeo, no Twitter timeline.
  • partialGoogle Fonts is third-party but doesn't set a cookie (it does leak IP to Google — see Floor E). Removing Google Fonts also closes this gap.
What to fix
  1. Decide on analytics posture before V11. Recommendation: if hosting on Vercel, enable Vercel Analytics (zero config, zero cookies, ~3 KB script). If hosting elsewhere, skip analytics entirely — the GitHub repo's traffic graph tells Danilo whether anyone's reading.
  2. If enabling any analytics, confirm no cookies set by opening DevTools → Application → Cookies on a fresh load. Banner must not appear.
  3. Close the Google Fonts leak (already covered in Floor E — but note it again, because it's also a privacy item, not just a perf item).

Floor HCode-craft floor · the repo is also the work sample #

What the floor is

Because github.com/mdanilorojas/dr-cv is public, the code is part of the portfolio. A reviewer will open it. They look for: meaningful commits, a README that explains the system, typed code, tests on critical paths, no dead code, no over-abstraction, no secrets.

Danilo's repo already shipped Phase 1 with 30 tests and a strong README thesis ("A system that keeps itself alive — with agents"). This section audits what's great, what's invisible, and what would move the needle for a tier-1 reviewer opening the repo in a new tab.

Specific test · the 10-item code-craft checklist
  1. H1README opens with a thesis, not a project description. First paragraph = what this is and why, not "to build run npm install".
  2. H2Architecture diagram or one-pager. Even ASCII. Reviewer should understand the system in < 30 seconds.
  3. H3Commit messages are Conventional Commits or similar disciplined format. Not "fix stuff", not "wip", not "updates".
  4. H4No any in TypeScript. tsc --strict --noEmit passes. Run on CI.
  5. H5Test suite is green and documented. npm test runs clean. README says how many tests and what they cover.
  6. H6CI runs on push. GitHub Actions: typecheck + tests + (optional) Lighthouse. Green badge in README.
  7. H7LICENSE file present (MIT, ISC, or CC-BY for portfolio content).
  8. H8No committed secrets or absolute paths. .env in gitignore. Grep for C:\ — no absolute user paths in committed files.
  9. H9No dead code or abandoned branches visible. dist/ either gitignored or intentionally committed with clear reason in README.
  10. H10Reviewer can run it in < 60 seconds. git clone && npm install && npm run build:all just works. No hidden env vars, no "you need to set up Supabase first".
