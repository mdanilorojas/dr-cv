Landing rebuild — aihero × dr-cv v11 merge.

Owner: Danilo Rojas Date: 2026-05-21 Visuals: browse mockups →

01 Why this exists

The current v11 landing reads as AI-generated. Two specific issues, called out by the user:

  1. Density. Hero · Proof · Horizon · Notebook · Work · Method · Contact = seven sections before the page releases pressure. Nothing breathes.
  2. Decorations. Chips, badges, segmented progress bar, tool rails, glow. Decoration competes with the work for attention.

Reference

aihero.dev — centered single column, editorial rhythm, content-first composition, restraint. The goal is to inherit that simplicity without changing the v11 design language.

02 Constraints kept

  • Tokens: design-system/tokens-v11.css is the source. No new tokens.
  • Type: JetBrains Mono display + Inter body stays. The mono headline is what makes the site feel like Danilo's, not someone else's editorial template.
  • Single accent: warm orange #FF8964 as punctuation only — one word in headlines, one CTA, hover states. Never as background fill, glow, or decorative chip.
  • Dark dominant: --v11-bg-primary for the whole page. No paper section, no alt-band rhythm.
  • EN + ES: language toggle in the nav.
  • Generator-driven: still rendered from data/*.yaml + generators/templates/v11-landing/. No hand-crafted HTML in dist/.

  ├── renderTestimonial     keep (v2, v3)

Data

  • data/horizon.yaml — kept on disk; consumed only by v3.

07 Success criteria

  • Visitor seeing it for the first time should not type "AI generated" or "template" in their head.
  • Hero thesis readable in one breath, no scroll required.
  • Click depth to contact intent: ≤ 1.
  • Build cost: under 200 LOC of new generator code, under 400 LOC of new CSS. The merge is a subtraction primarily.
  • All data/*.yaml content keeps rendering — no copy is orphaned without an explicit decision.
  • Builds via existing npm run build:landing-v11, no new dependencies.

08 Open questions

Resolve before plan

  1. Notes source. v2 has a notes block. New data/notes/*.md directory? Or drop notes from v2 and recommend v1?
  2. Case detail pages. Current generator produces /work/<slug>/ case detail. Survive (linked from each work item) or collapse into expand-on-click?
  3. Language toggle. Current SPA toggles in-place. New static page either reloads to /es/ or keeps SPA toggle. Recommendation: static reload.
  4. Final iteration target. v2 is recommended. Plan will assume v2 unless told otherwise.

09 Out of scope

  • Case detail page redesign (separate spec).
  • CV / skills sheet generators.
  • Animations beyond CSS hovers.
  • Light-mode variant.
